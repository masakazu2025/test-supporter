# プロジェクト ロードマップ

## 現在のフェーズ

**スケルトンフェーズ** — UI構造・モジュール間インターフェースの確認

各機能のスケルトン（バックエンド: ルーター+型定義、フロントエンド: 画面レイアウト+モックデータ）を
先に通しで作り、結合点での問題を早期に発見する。

---

## 機能別進捗

| 機能 | 状態 | 概要 |
|------|------|------|
| **採取** | 🚧 進行中 | リモート端末からファイルをプールに取得（Phase2完了） |
| **端末（閲覧・検索）** | 🔲 スケルトン待ち | エントリ/画像/ログの閲覧・検索 |
| **評価** | 🔲 スケルトン待ち | テストケース・エビデンス・不具合管理 |
| **設定** | 🔲 スケルトン待ち | 顧客プロファイル・端末マスタ管理 |

## スケルトンフェーズの順序

```
1. 設定   ← 他機能の基盤（端末マスタがないと動かない）
2. 端末   ← 採取インフラの上に乗る閲覧・検索
3. 評価   ← 端末・設定が揃ったうえで設計
```

各機能で作るもの：

| 成果物 | 内容 |
|--------|------|
| `docs/features/{機能}/requirements.md` | 要件（随時更新） |
| バックエンド: ルーター + レスポンス型 | 実装は 200 固定返し |
| フロントエンド: 画面レイアウト | モックデータで表示 |

---

## 依存関係メモ

- infra層（`file_transfer`, `archive`, `unc_path` 等）は採取・閲覧・仕訳が共有する
- プール（ローカルキャッシュ）は全機能の共通基盤
- ファイル解決レイヤー（プール優先→端末フォールバック）は評価・閲覧が共用する
- DBスキーマ（`docs/schema/db.md`）は採取 Phase4 で確定する
