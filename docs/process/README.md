# AIエージェントを活用した開発プロセス設計

## 目的

人間とAIエージェントが対話しやすく、修正しても品質が維持できる開発サイクルを構築する。
無限期間ではなく、有限期間の品質維持を現実的なゴールとする。

## 基本構造

```
要件 → 初期設計 → I/F定義 → 詳細設計 → コーディング  （下向き）
要件 ← 初期設計 ← I/F定義 ← 詳細設計 ← コーディング  （上向き：やりにくさが矛盾を露わにする）
                               ↑
                         横にいる人間（全体整合の判断）
```

各フェーズに異なるエージェントを割り当て、意識（人格）を安定させる。
横にいる人間がフレームワーク自体の妥当性を判断し、その視点を言語化することでエージェントに人格を与える。

## ソフトウェア変換可能性

抽象的な概念をソフトウェアに変換するとは、**観測可能な状態変化として記述できること**。

### 変換の連鎖

```
概念（志向性が曖昧）    例: 管理する
  ↓ 操作化
操作（一意に解釈できる） 例: コピーする・登録する
  ↓ 形式化
関数名・モジュール名     例: copy_to_staging() / register_to_db()
```

### 操作化の判断基準

| 基準 | 説明 |
|------|------|
| **具体的な一動詞に収まるか** | 抽象動詞（管理・処理・対応）は概念が残っているサイン |
| **戻り値が言えるか** | 戻り値が特定できる = 操作化の完了チェックポイント |
| **テストのアサートが書けるか** | 書けない → 分割不足 / 実装依存 → 分割しすぎ |

### 戻り値の設計方針

- **boolで終わる操作**：成否だけわかればよい。インターフェースの議論を深くしなくてよい
- **具体的な値を返す操作**：次の操作への入力になる。モジュール間の契約として厳密に定義する

この区別がレビューコストの配分と、人間・AIの役割分担の基準にもなる。

### BDDとの接続

```
Given: 初期状態（入力の前提）
When:  操作（具体的な一動詞）
Then:  戻り値・状態変化（観測可能な完了状態）← ここが書けないなら詳細設計が未完
```

### インターフェース定義の位置づけ

戻り値は受け取る側が何を必要とするかで決まる。そのためI/F定義は初期設計と詳細設計の間に来る。

```
初期設計     : モジュールを特定し、依存の方向を決める
I/F定義      : 戻り値の型・構造をモジュール間で合意する（契約）
詳細設計     : 各モジュール内の振る舞いをBDDで記述する
```

## 議題と優先順位

| 優先 | ファイル | 内容 |
|------|----------|------|
| 1 | `phases.md` | 各フェーズの定義・境界条件・上向きフィードバックのトリガー |
| 2 | `agent-identities.md` | 各エージェントの人格定義・見てよい範囲/見てはいけない範囲 |
| 3 | `phase-initial-design.md` | 初期設計フェーズの詳細（最も議論が必要な箇所） |
| 4 | `overview.md` | 目的・双方向フロー・横にいる人間の役割の言語化 |

## 移行方針

採取機能の実装が完了するまでは現状の開発フローで進める。
実装を通じて構想を検証しながら、完了後に徐々にこのフレームワークへ移行する。

## 未解決の問い

- 初期設計と詳細設計の移行条件をどう形式化するか
- 初期設計エージェントが「実装思考」に引っ張られないための制約定義
- エージェントのランク（フェーズ内／フェーズ間／フレームワーク全体）の設計
- 過剰分割を防ぐ分割タイミングの判断基準（戻り値基準で概ね解決しているが要検証）
