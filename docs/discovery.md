### 端末監視対象選択画面
対象となる端末のIP一覧を並べる。対象日付と、開始・停止ボタンを用意
開始を押下することで、対象端末を定期的に監視し、指定したファイルの同期を行う
同期の方法は、エントリをキーとして、それに紐づくSS、ログ、追加採取ファイルなどをひとつのフォルダにまとめて採取していく

#### 課題・検討事項

**採取ロジックの拡張**（方針決定）

採取と仕訳を完全に分離する設計に移行する。

```
採取（常に自動・バックグラウンド）
  └─ エントリ：全件同期
  └─ 画像：全件同期（エントリフォルダには格納しない → 後述）
        ↓
仕訳（ユーザーがストラテジーを選ぶ）
  ├─ タイムスタンプ範囲で自動紐づけ
  ├─ 一覧から手動選択
  └─ （将来）エントリNo範囲指定 など
```

**同期あり / なしの選択**
* 同期なし（従来の方式3相当）：仕訳時にリモート端末へ直接問い合わせ
* 同期あり（方式4相当）：ローカルプールから取得
* 仕訳ストラテジーはデータソース（ローカル/リモート）を意識しない設計とする

**これにより解決されること**
* 3と4の採取ロジックを1本に統一できる
* 仕訳ストラテジーは後から追加可能（採取ロジックに触らない）
* ユーザーが状況に応じてストラテジーを選べる

**端末の日付変更への追随**
* 監視はIP＋日付の組み合わせでユニーク登録している
* 端末の日付が変更されることがあり、その変更に追随する仕組みが必要

**過去日付・一括同期**
* 過去の日付のエントリを取得したい
* または、端末と一括で同期する機能

**並列ダウンロードの方針**（決定：whileループ方式を維持）
* 採取ロジックを関数として分離し、whileループから呼び出す形で実装する
* 手動DL・一括同期はAPIエンドポイント経由でasyncタスクを追加起動するだけでよい
* 採取ロジックが関数で分離されていれば、後からタスクキュー方式への移行も容易

> **アイデア（将来の選択肢）：タスクキュー方式**
> 採取ロジックをタスクキュー（asyncio.Queue + ワーカー）で管理する方式。
> メリット：DL状態の可視化、失敗タスクの自動リトライ、手動DLの自然な割り込み。
> 実装：FastAPI + SQLite（jobsテーブル）で外部ミドルウェア不要で実現可能。
> 採取ロジックが関数で分離されていれば、orchestration部分を差し替えるだけで移行できる。

**LAN切断・復帰時の問題**
* LAN切断を含む試験で、復帰後にDLが再開されないことがある
* 接続状態や同期状態を可視化する仕組みがほしい

**エントリの一覧表示と一括保存**

#### 中央ペイン
選択したアイテムの情報を表示する
* テキスト形式
* 表形式
  * 列をテキスト形式でコピーするボタン
  * フィルタ（列をフィルタすることを可能とする）
    * 一括フィルタ：貼り付けたデータを正規化して、対象Noを一括してフィルタ
      * 正規化ルール：数字以外の文字を除去 → 半角に統一
      * 背景：テスト仕様書のNoの書き方が人によって異なる（全角/半角、数字のみ/「No.1」のような接頭詞あり）
  * フィルタの状態は保持する
* json形式
* 上記表示内容の下部に、ペジネーションを設置
  * <<、<、[番号入力]、>、最新 ：最初・前・番号指定・次・最新エントリへ移動
  * 開く：保存したフォルダをエクスプローラで開く
  * 保存：右側に仕訳ペインを表示・非表示
  * 不具合：右側に不具合保存用ペインを表示・非表示

#### 右ペイン
##### 選択したエントリを仕訳るためのペイン
仕訳先フォルダの構造：`機能名 / シート名 / 任意フォルダ（省略可・複数階層可） / エントリ`

* 機能名（ドロップダウン）
* シート名（ドロップダウン）（Excelから読み込んだシート名）
* 任意フォルダ（省略可・複数指定可）
  * ラジオボタンで「項番」または「任意名称」を選択
  * 項番の場合
    * +/- ボタンで番号を増減
    * 手入力時は数値のみ入力可能
  * 任意名称の場合
    * 自由テキスト入力
* アクションボタン
  * 保存：指定したフォルダ構造にエントリを保存
  * 削除：仕訳済み保存先の、現在表示中のエントリNoと同一のエントリフォルダのみ削除（採取元は削除しない）
  * 開く：保存先フォルダをエクスプローラで開く

##### 不具合保存用ペイン（現状・暫定仕様）
不具合発生時に提出するファイル群を揃えるための機能。

* 収集対象ファイルは設定画面で事前に定義しておく
* 保存先の指定：項番または任意名称
* 保存ボタン押下で、設定済みのフォルダ・ファイルを収集してコピー保存
  * エントリファイルも合わせてコピー
  * コピー元は仕訳済み保存データ（採取元ではない）

> **制約・経緯：** 現状はエントリ単位での保存に固定されている。
> SSのみ・ログのみ など、提出する資料の組み合わせが案件によって異なるため、
> 時間的制約により最低限の実装にとどめた暫定仕様。

### 設定画面
顧客プロファイルごとに以下を管理する。詳細は実装しながら詰める。

* 接続先端末の設定（IP等）
* Excelテスト仕様書の読み込み・シート名の取得（シートの追加・並び替えが可能）
* 保存先のルートパス
* 不具合採取対象ファイルの定義
* 項目名定義ファイルの生成（設計書を読み込み、閲覧時の項目名を定義したJSONを生成）

**sync_targets（採取ターゲット定義）**（方針決定・更新済み）

`pool_root` は廃止。`dst_dir` にフルパス（絶対パス）を書く。
`dst_dir` が `\\` 始まりならUNCパス（リモート操作）、それ以外はローカル操作と判定する。

```json
{
  "archiver_path": "C:\\Tools\\myarchiver.exe",
  "sync_targets": [
    {
      "type": "entries",
      "action": "copy",
      "src_dir": "C:/TestApp/entries",
      "pattern": ".*\\.gz",
      "dst_dir": "D:\\TestPool\\entries\\{ipaddress}\\{yyyymmdd}",
      "rename": null,
      "watch": "realtime",
      "post_process": { "decompress": "gz" }
    },
    {
      "type": "screenshots",
      "action": "copy",
      "src_dir": "C:/TestApp/screenshots",
      "pattern": ".*\\.png",
      "dst_dir": "D:\\TestPool\\screenshots\\{ipaddress}\\{yyyymmdd}",
      "rename": "{ipaddress12}_{timestamp}.{ext}",
      "watch": "realtime",
      "post_process": { "decompress": null }
    },
    {
      "type": "screenshots_weekly_archive",
      "action": "move",
      "src_dir": "C:/TestApp/screenshots",
      "pattern": ".*\\.png",
      "dst_dir": "\\\\{ipaddress}\\C$\\TestApp\\screenshots\\{yyyymmdd}",
      "rename": "{ipaddress12}_{timestamp}.{ext}",
      "schedule": "Friday 19:00",
      "post_process": { "decompress": null }
    }
  ]
}
```

**テンプレート変数**

| 変数 | 解決方法 |
|---|---|
| `{ipaddress}` | 監視セッションの端末IP（そのまま） |
| `{ipaddress12}` | IPを12桁数字に変換（例: `192.168.1.2` → `192168001002`） |
| `{yyyymmdd}` | 各ファイルの mtime の日付部分 |
| `{timestamp}` | 各ファイルの mtime の日時（`yyyymmddHHMMSS`） |
| `{ext}` | 元のファイル拡張子 |

**各フィールドの意味**

| フィールド | 説明 |
|---|---|
| `action` | `copy`（元ファイル保持）/ `move`（元ファイル削除） |
| `src_dir` | リモート端末上のWindowsパス（UNC変換して読み取る） |
| `pattern` | 対象ファイルの正規表現フィルタ |
| `dst_dir` | 保存先のフルパス（テンプレート）。`\\` 始まりはUNC（リモート操作） |
| `rename` | リネーム規則テンプレート。`null` はリネームなし |
| `watch` | `realtime`（毎ループ）/ `interval`（N分ごと） |
| `schedule` | 時刻指定トリガー（`watch` と排他） |
| `post_process.decompress` | 解凍方式。`gz` = 専用アーカイバ（subprocess）、`null` = なし |

**UNCパス変換**

`src_dir` は端末上のWindowsパス。アクセス時に自動でUNCに変換する。

```
C:\TestApp\entries  +  terminal=192.168.1.10
    →  \\192.168.1.10\C$\TestApp\entries
```

### 採取アーキテクチャ（方針決定）

#### プール＝キャッシュ思想

```
リモート端末
    ↓ 採取（自動 or 手動）
ローカルプール（キャッシュ）← 全機能の共通格納先
    ↓ 優先して参照
仕訳 / 不具合採取
    ↓ プールに不足があれば
リモート端末（フォールバック）
```

* 採取はプールに貯めるだけ。仕訳・閲覧はプールから読む
* 個別採取（エントリNo指定など）も同じプールに保存する
* 仕訳と不具合採取は「データソースを抽象化」して実装する（ローカル/リモートを意識しない）

#### セレクター抽象化

採取パイプライン（`sync_targets` で定義）は共通。「どのファイルを対象にするか」をセレクターで切り替える。

| パターン | セレクター |
|---|---|
| 自動採取（watch） | `{ "type": "all_new" }` |
| エントリNo指定 | `{ "type": "entry_id", "ids": ["00001"] }` |
| エントリNo範囲 | `{ "type": "entry_range", "start": "00001", "end": "00010" }` |
| タイムスタンプ範囲 | `{ "type": "time_range", "start": "...", "end": "..." }` |
| 不具合採取 | time_range または entry_range（＋ログ直採取を追加） |

#### 機能ごとのデータソース

| 機能 | 主なデータソース | 特記 |
|---|---|---|
| 自動採取 | リモート → プール | バックグラウンド常時 |
| 手動採取（1〜3） | リモート → プール | 同じパイプライン |
| 仕訳 | プール（→ フォールバック） | - |
| 不具合採取 | プール（→ フォールバック）＋ ログ直採取 | ログのみ例外 |

#### エントリの採取フロー（確定）

```
アトミックコピー:  00001.gz.tmp → 00001.gz
アトミック解凍:    00001.tmp/   → 00001/      ← 採取完了の証明
DB記録:            解凍完了後に記録（ジョブ管理には使わない）
アーカイブ保持:    00001.gz を削除しない
```

リトライ判定はファイルシステムの状態のみで完結（DBは不要）。

#### スクリーンショットのリネーム（確定）

```
リモートの mtime から: {ipaddress12}_{yyyymmddHHMMSS}.{ext}
例: 192168001010_20260228143052.png
```

### SS採取画面
リモート端末を直接参照し、取得した画像を保存する機能。

* 画像一覧を表示し、単独または一括で選択して保存
* 保存先のフォルダ構造はエントリ保存と基本的に同じ考え方
* 画像と一緒に保存するファイルを指定可能
  * 任意ではなく、PG固定の候補から選択する方式

> **UI課題：** 現状はエントリ保存とインターフェースが異なる。統一した保存UIに整理したい。

#### SS管理の方針（暫定案）

**画像はエントリと独立して同期する**（方針決定）
* 画像はエントリフォルダに格納しない。独立したローカルプールとして同期する
* エントリとの紐づけは仕訳時に行う（採取時点では紐づけない）
* これにより「画像のみ閲覧」機能が実現可能になる

**自動採取時の画像タイムスタンプ問題への対策**
自動採取（前エントリ～現エントリのタイムスタンプ計算）で不要な画像混入・必要な画像漏れが発生する問題への対処：

| 対策 | 方針 | 優先度 |
|------|------|--------|
| バッファ設定（固定値） | 設定画面で開始・終了それぞれにバッファ設定可能 | ベースライン（実装済み） |
| 仕訳時の画像確認・選択 | 仕訳UIでサムネイル表示し、不要なものを外す。デフォルトは自動選択済み | 高 |
| 仕訳済みデータの事後編集 | 仕訳後に間違いに気づいた場合のアプリ内修正機能 | 必須（別軸） |
| 画像認識による自動判断 | コスト対効果が合わない | 見送り |
| エントリ内容に応じたバッファ延長 | 特定エントリ情報が含まれる場合にバッファを動的に変更 | 後回し |

**基本：ローカル同期方式**
* 監視whileループの中で、SSフォルダを定期的に差分同期してローカルに保存する
* 閲覧・仕訳はローカルの同期済みデータを使う（高速・接続断でも閲覧可能）

**リアルタイム確認：オンデマンド取得**
* テスト実施中など「今すぐ見たい」場面では、更新ボタンで端末から直接フェッチする
* 取得した画像はそのままローカルにも保存し、同期データと統合する

**一覧の更新タイミング**
* 定時（N秒ごとの自動更新）・手動（更新ボタン）・イベント（採取完了後）の3トリガーがある
* いずれも「一覧を最新にする」という同じ処理であり、共通関数として実装する

```
# 定時はwhileから呼ぶ
while 監視中:
    sync_screenshots()   ← 共通関数
    await asyncio.sleep(N)

# 手動・イベントはAPIから呼ぶ
POST /api/screenshots/sync → sync_screenshots()
```

* 重複実行（更新中に更新が走る）だけ自前で制御する
* 状態の可視化・リトライ管理が必要になった場合はタスクキュー方式への移行を検討（→ 端末監視のタスクキューアイデアと同様）

### 保存インターフェースの統一
以下の3つの保存操作が、現状それぞれ異なるUIになっている。共通化を検討したい。

| 保存操作 | 現状 |
|---------|------|
| エントリ保存（仕訳ペイン） | 独自UI |

| エントリ保存 ＋ 追加ファイル指定 | 未実装（ニーズあり） |

* エントリ保存時に、その試験だけ追加でファイルを保存したいシチュエーションがある
* 保存先フォルダの構造の考え方は共通なので、統一した保存コンポーネントにまとめられるはず
