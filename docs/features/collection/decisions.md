# 採取 設計の意思決定

## 1. プール＝キャッシュ思想

**決定：** ローカルプールを全機能共通のキャッシュとして使う。

**背景：**
- 仕訳・閲覧・不具合採取が同じリモートファイルにアクセスする必要がある
- 毎回リモートから取得すると遅く、LAN切断時に使えない

**結果：**
- 採取と仕訳を完全に分離できる
- オフラインでも仕訳・閲覧が動作する
- 採取機能がプールに書くだけで、他機能はプールから読むだけになる

---

## 2. ファイルシステムが状態を管理する（DBはジョブ管理に使わない）

**決定：** `.tmp` 経由のアトミックリネームでコピー完了を管理する。DBは採取完了後の記録のみ。

**背景：**
- ジョブキューをDBで管理するとFS状態とDBの二重管理になり、ズレが生じる
- `.tmp` ファイルの存在で「コピー中断」を検知でき、シンプルに再開できる

**結果：**
- リトライ判定がFSだけで完結する（`{entry_no}/` あり → スキップ、`.tmp` あり → 削除して再開）
- DBはクエリ高速化のためだけに使う

---

## 3. Sync はAutoWatcherだけの概念

**決定：** `CopyJob` と Worker は "Sync" を知らない。

**背景：**
- 手動採取は全件同期（Sync）ではなく、指定ファイルのコピー（Targeted Copy）
- 「メイン端末は自動採取、比較用端末は特定エントリのみ手動採取」のユースケースがある

**結果：**
- `CopyJob` は `{ terminal, src_file, target }` のシンプルな構造になる
- Worker は「コピーして後処理してDB登録する」だけに専念できる
- AutoWatcher と手動採取API が同じキューに Job を積むだけになる

---

## 4. field_extractor は Worker 専用

**決定：** AutoWatcher・手動採取セレクターは `field_extractor` を使わない。

**背景：**
- AutoWatcher は速度優先。ファイル拡張子フィルタ程度で十分
- 手動採取の絞り込みは UI の field 値から pattern を生成して行う
- field_extractor はコピー後の DB 登録のためのメタデータ抽出が役割

**結果：**
- AutoWatcher がシンプルになる（pattern マッチのみ）
- Worker が単一の責務（コピー・抽出・DB登録）を持つ

---

## 5. 端末ごとに Worker スレッドを1本

**決定：** 端末ごとに deque + Worker スレッドを持つ（2〜3台並列）。

**背景：**
- 複数端末を同時採取したいが、1端末内の順序は保証したい
- ファイル単位の細かい並列化はLAN環境では不要

**結果：**
- 端末A が重くても端末B は進む
- 端末内の処理順序が保証される（キューの順序通り）
- スレッド数が端末数に比例するため管理しやすい

---

## 6. src_dir は採取ソースが解決する

**決定：** Worker はリモートのファイル一覧取得・UNC変換を行わない。採取ソースが解決済みの `src_file`（UNCパス）を渡す。

**背景：**
- 「何を採取するか」の決定は AutoWatcher（全件）や手動採取（指定）の責務
- Worker が「どこから取るか」まで知ると責務が広がりすぎる

**結果：**
- Worker がシンプルになりテストしやすい
- AutoWatcher と手動採取が独立してファイル選択ロジックを持てる

---

## 7. infra 層の分割方針

**決定：** 読み取り（`file_io`）と転送（`file_transfer`）を分離する。

**背景：**
- `file_io`（読み取り）は採取・閲覧・仕訳すべてで使う汎用処理
- `file_transfer`（copy/move）は採取・仕訳で使う転送処理
- 用途・副作用の有無が異なる

**結果：**
- `file_transfer` は副作用ありの処理として明確に分離できる
- `file_io` は閲覧機能でも再利用できる
