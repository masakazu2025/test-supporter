# 採取 要件

## 概要

リモート端末上のファイルをローカルプールに取得する機能。
採取はデータを集めるだけで、分類・保存先の決定（仕訳）は行わない。

## ユースケース

| # | 採取方法 | 説明 | トリガー |
|---|---------|------|---------|
| 1 | **自動採取** | 端末を定期監視し、新着ファイルを自動取得 | 監視開始ボタン |
| 2 | **手動採取（エントリ指定）** | エントリNo を指定してDBを検索。未採取の場合はリモートから取得 | UI操作 |
| 3 | **手動採取（範囲指定）** | エントリNo の範囲を指定してDBを検索。未採取の場合はリモートから一括取得 | UI操作 |
| 4 | **手動採取（時刻範囲）** | 開始〜終了日時でDBを検索。未採取の場合はリモートから取得 | UI操作 |

### 手動採取の基本方針

手動採取は **DB（pool）を一次検索ソースとし、ミス時のみリモートへフォールバック** する。

```
UI（検索条件入力）
  ↓
DB検索（poolのmetadata）
  ├─ ヒット → poolのファイルパスを返却（採取不要）
  └─ ミス  → リモートから該当ファイルを特定 → enqueue（priority=HIGH）
```

- プール=キャッシュ思想に基づく（すでに採取済みのものは再採取しない）
- field_extractor はWorkerがコピー時に使うため、検索時には不要

## 採取対象

| 種別 | 説明 | 後処理 |
|------|------|--------|
| `entries` | エントリアーカイブ（.gz） | 専用アーカイバで解凍・解凍後もアーカイブ保持 |
| `screenshots` | スクリーンショット画像 | ファイル名をmtimeベースにリネーム |
| `files` | 任意ファイル | なし |

## 採取先

- ローカルプール（全機能共通のキャッシュ）
- 採取先パスは `CollectionTarget.dst_dir` テンプレートで決定（設定ファイルで定義）

## 並列処理

- 複数端末を同時採取できる（2〜3台を想定）
- 1端末内は順次処理

## 優先度制御

- 手動採取はキューの先頭に挿入（自動採取より優先して処理される）

## 監視方式

| 方式 | 説明 |
|------|------|
| `realtime` | 監視ループのたびに同期 |
| `interval` | N分ごとに同期 |
| `schedule` | 時刻指定で実行（例: 毎週金曜19時） |
